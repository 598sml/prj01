# Peer Feedback Form

After each presentation, you will provide short written feedback to the presenter. The goal is to help them improve their project before they invest significant time in it.

The goal is not to judge the project, but to help make it clearer.

Your feedback should be
* Specific – Avoid vague comments like “seems good.”
* Constructive – Focus on helping the project improve.
* Honest – If something seems unclear, say so.
* Actionable – Suggest something the presenter could actually do.

You will first write your feedback silently, then we will briefly discuss it as a group.

### 1. My understanding of the project

In one or two sentences, describe what you think the project is about.

Example:

“I understand this project to be comparing FNO and PINNs for learning solutions to the 1D Burgers equation using simulated training data.”

This helps the presenter see whether their explanation was clear.

### 2. One strength of this project

What seems promising or well thought-out?

Examples might include:
	•	A clear and interesting scientific question
	•	A good reference paper to follow
	•	A realistic implementation plan
	•	A strong validation strategy
	•	Good use of available data or simulators

Be specific if possible.

### 3. One concern or potential risk

What seems unclear, risky, or potentially too ambitious?

Examples might include:
	•	The scope seems too large
	•	Data availability is uncertain
	•	The implementation plan is vague
	•	The evaluation strategy is unclear
	•	The project may rely too heavily on existing code

The goal is to identify the biggest issue that might cause problems later.

### 4. One concrete suggestion

Provide one specific suggestion that could improve the project.

Examples:
	•	Suggest a simpler baseline method
	•	Suggest a dataset or benchmark
	•	Recommend narrowing the scope
	•	Suggest a clearer evaluation metric
	•	Recommend a paper or implementation to look at

Try to make your suggestion specific and actionable.
